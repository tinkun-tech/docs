    OIDC Authentication¶

    This guide covers the core OIDC (OpenID Connect) authentication functionality for user authentication and session management.

    Overview¶

    The authentication module provides:

    • User authentication via Okta OIDC provider
    • Authorization code flow with PKCE support
    • State and nonce validation (CSRF and replay attack prevention)
    • Automatic user creation and profile management
    • Cookie-based session management with secure HTTP-only cookies
    • JWT token verification

    Authentication Flow¶

    The complete OIDC authentication flow consists of several steps:

    1. Initiate Authentication¶

    # Client calls the authenticate endpoint
GET /api/oidc/authenticate/?login_hint=user@example.com

    The backend: - Generates a unique state parameter for CSRF protection - Generates a nonce for replay attack prevention (if enabled) - Creates PKCE code verifier and challenge (if enabled) - Returns authorization URL for Okta redirect

    2. User Authentication at Provider¶

    The user is redirected to Okta where they: - Enter their credentials - Complete any MFA requirements - Authorize the application

    3. Authorization Callback¶

    Okta redirects back to your frontend with an authorization code:

    http://localhost:3000/authorization-code/callback?code=xxx&state=yyy

    4. Token Exchange¶

    Your frontend sends the code to the callback endpoint:

    # POST /api/oidc/callback/
{
    "code": "authorization_code_from_okta",
    "state": "state_from_url_params"
}

    The backend: - Validates state matches the one stored in cookies - Exchanges authorization code for tokens (access, ID, refresh) - Verifies JWT tokens using Okta's public keys - Creates or updates the user in your database - Sets secure HTTP-only cookies with the tokens - Returns user information

    5. Authenticated Requests¶

    Once authenticated, the user can make API requests:

    # GET /api/oidc/user/
# Cookies are automatically sent by the browser

    The authentication system: - Validates access token from cookies - Automatically refreshes expired tokens using refresh token - Updates cookies with new tokens transparently

    User Management¶

    Automatic User Creation¶

    By default, users are automatically created on first login:

    # settings.py
TINKUN_OIDC = {
    "OIDC_CREATE_USER": True,  # Default
}

    User information is extracted from the ID token claims: - sub - Unique user identifier - email - User's email address - Additional claims based on your Okta configuration

    Custom User Service¶

    Override the default user service for custom user creation logic:

    # myapp/services.py
from django.contrib.auth import get_user_model

from tinkun_oidc.services.user_service import OIDCUserService as BaseUserService

User = get_user_model()


class CustomUserService(BaseUserService):

    def create_user(self, claims: dict):
        """Custom user creation with additional fields"""
        user = super().create_user(claims)

        # Add custom fields
        user.first_name = claims.get("given_name", "")
        user.last_name = claims.get("family_name", "")
        user.profile.department = claims.get("department", "")
        user.save()

        return user

    def update_user(self, user, claims: dict):
        """Update user information on each login"""
        user.email = claims.get("email")
        user.first_name = claims.get("given_name", "")
        user.last_name = claims.get("family_name", "")
        user.save()

        return user

    Configuration:

    TINKUN_OIDC = {
    "OIDC_USER_SERVICE": "myapp.services.CustomUserService",
}

    Disable User Creation¶

    If you only want to authenticate existing users:

    TINKUN_OIDC = {
    "OIDC_CREATE_USER": False,
}

    Users that don't exist in your database will receive an authentication error.

    Security Features¶

    State Parameter¶

    Protects against CSRF attacks by: - Generating a cryptographically random state on authentication request - Storing it in a secure HTTP-only cookie - Validating it matches during callback

    Configuration:

    TINKUN_OIDC = {
    "OIDC_STATE_SIZE": 32,  # Size in bytes (default)
}

    Nonce¶

    Prevents replay attacks by: - Generating a unique nonce for each authentication request - Including it in the ID token request - Validating the nonce in the returned ID token

    Configuration:

    TINKUN_OIDC = {
    "OIDC_USE_NONCE": True,     # Enable nonce (default)
    "OIDC_NONCE_SIZE": 32,      # Size in bytes (default)
}

    To disable:

    TINKUN_OIDC = {
    "OIDC_USE_NONCE": False,
}

    PKCE (Proof Key for Code Exchange)¶

    Protects authorization code flow by: - Generating a random code verifier - Creating SHA-256 hash as code challenge - Sending code challenge with authorization request - Sending code verifier during token exchange

    Configuration:

    TINKUN_OIDC = {
    "OIDC_USE_PKCE": True,                 # Enable PKCE (default, recommended)
    "OIDC_CODE_CHALLENGE_METHOD": "S256",  # SHA-256 hashing (required)
}

    Note: PKCE is enabled by default following OAuth 2.0 security best practices and should not be disabled.

    Cookie-Based Token Storage¶

    Tokens are stored in secure HTTP-only cookies to prevent XSS attacks:

    Cookie Purpose Expiration
    TKNA Access token for API authentication Session
    TKNI ID token with user identity Session
    TKNR Refresh token for obtaining new tokens 4 days (configurable)
    TKNS State parameter for CSRF protection Session
    TKNN Nonce for replay protection Session
    TKNV PKCE code verifier Session

    Temporary Cookies During Refresh¶

    When tokens are refreshed, temporary cookies are used:

    Cookie Purpose
    TKNTA New access token (replaced after response)
    TKNTI New ID token (replaced after response)
    TKNTR New refresh token (replaced after response)

    Custom Cookie Names¶

    TINKUN_OIDC = {
    "OIDC_ACCESS_TOKEN_COOKIE_NAME": "my_access_token",
    "OIDC_ID_TOKEN_COOKIE_NAME": "my_id_token",
    "OIDC_REFRESH_TOKEN_COOKIE_NAME": "my_refresh_token",
}

    Cookie Expiration¶

    TINKUN_OIDC = {
    "OIDC_REFRESH_TOKEN_COOKIE_MAX_AGE": 60 * 60 * 24 * 7,  # 7 days
}

    JWT Verification¶

    All tokens are verified using Okta's JWT verification:

    ID Token Verification¶

    Required claims in ID token:

    TINKUN_OIDC = {
    "OIDC_ID_TOKEN_CLAIMS": ("iss", "exp", "iat", "sub", "email"),
}

    Access Token Verification¶

    Required claims in access token:

    TINKUN_OIDC = {
    "OIDC_ACCESS_TOKEN_CLAIMS": ("iss", "aud", "exp", "uid"),
}

    Custom JWT Verifier¶

    Override for custom verification logic:

    # myapp/services.py
from tinkun_oidc.services.jwt_service import OIDCJWTVerifierService as BaseJWTService


class CustomJWTVerifier(BaseJWTService):

    def verify_access_token(self, token: str) -> dict:
        """Custom access token verification"""
        claims = super().verify_access_token(token)

        # Add custom validation
        if claims.get("scp") and "admin" not in claims["scp"]:
            raise ValueError("User requires admin scope")

        return claims

    Configuration:

    TINKUN_OIDC = {
    "OIDC_JWT_VERIFIER_SERVICE": "myapp.services.CustomJWTVerifier",
}

    Logout¶

    Backend Logout¶

    # POST /api/oidc/logout/

    The backend: - Revokes the refresh token with Okta - Clears all authentication cookies - Returns the Okta logout URL

    Frontend Integration¶

    // 1. Call backend logout endpoint
const response = await fetch('/api/oidc/logout/', {
    method: 'POST',
    credentials: 'include'
});

const data = await response.json();

// 2. Redirect to Okta logout URL
window.location.href = data.logout_url;

    Examples¶

    Basic Setup¶

    # settings.py
TINKUN_OIDC = {
    "OKTA_DOMAIN": "your-domain.okta.com",
    "OIDC_RP_CLIENT_ID": "your-client-id",
    "OIDC_RP_CLIENT_SECRET": "your-client-secret",
}

    Custom Scopes¶

    TINKUN_OIDC = {
    "OKTA_DOMAIN": "your-domain.okta.com",
    "OIDC_RP_CLIENT_ID": "your-client-id",
    "OIDC_RP_CLIENT_SECRET": "your-client-secret",
    "OIDC_RP_SCOPES": "openid email profile groups offline_access",
}

    Production Settings¶

    import environ

env = environ.Env()

TINKUN_OIDC = {
    "OKTA_DOMAIN": env("OKTA_DOMAIN"),
    "OIDC_RP_CLIENT_ID": env("OIDC_RP_CLIENT_ID"),
    "OIDC_RP_CLIENT_SECRET": env("OIDC_RP_CLIENT_SECRET"),
    "OIDC_AUTHENTICATION_CALLBACK_URL": env("OIDC_CALLBACK_URL"),
    "OIDC_USE_NONCE": True,
    "OIDC_USE_PKCE": True,
    "OIDC_CREATE_USER": True,
}
